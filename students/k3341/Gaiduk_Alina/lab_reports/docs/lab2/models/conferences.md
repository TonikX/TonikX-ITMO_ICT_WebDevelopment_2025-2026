# Модели приложения `conferences`

В данном разделе описаны основные модели, используемые в приложении `conferences`. Эти модели реализуют предметную область «научные конференции» в соответствии с заданием лабораторной работы.

## 1. Модель `Topic`

```
class Topic(models.Model):
    name = models.CharField(max_length=100, unique=True)

    class Meta:
        ordering = ["name"]

    def __str__(self):
        return self.name
```

**Назначение.**
`Topic` описывает отдельную тематику конференции (например, "машинное обучение", "кибербезопасность").
Темы нужны для того, чтобы каждая конференция могла быть привязана к нескольким научным направлениям.

**Особенности:**

* `name` — уникальное название тематики.
* `ordering = ["name"]` — все объекты темы по умолчанию сортируются в алфавитном порядке.
* `unique=True` не позволяет создать дубликат одинаковой темы.

Таким образом, модель `Topic` используется как справочник тематик.

---

## 2. Модель `Place`

```
class Place(models.Model):
    name = models.CharField(max_length=200)
    address = models.CharField(max_length=255)
    description = models.TextField()

    class Meta:
        ordering = ["name"]
        unique_together = ("name", "address")

    def __str__(self):
        return self.name
```

**Назначение.**
`Place` описывает место проведения конференции.

**Сохраняемые данные:**

* `name` — название площадки / вуза / центра.
* `address` — адрес места проведения.
* `description` — описание площадки (например, инфраструктура, аудитории, условия размещения).

**Особенности:**

* `unique_together = ("name", "address")` запрещает в базе два одинаковых места с тем же названием и адресом.
* `ordering = ["name"]` сортирует площадки по названию.

Модель `Place` позволяет повторно использовать одну и ту же площадку для разных конференций.

## 3. Модель `Conference`

```
class Conference(models.Model):
    title = models.CharField(max_length=200)
    topics = models.ManyToManyField(Topic)
    place = models.ForeignKey(Place, on_delete=models.CASCADE, null=True)
    start_date = models.DateField()
    end_date = models.DateField()
    description = models.TextField()
    participation_terms = models.TextField(blank=True, default="")

    class Meta:
        ordering = ["-start_date"]

    def __str__(self):
        return self.title
```

**Назначение.**
`Conference` — центральная сущность системы. Она описывает конкретную научную конференцию.

**Сохраняемые данные:**

* `title` — название конференции.
* `topics` — связи «многие ко многим» с моделью `Topic`. Это даёт возможность указывать несколько тематик.
* `place` — ссылка на `Place`, где проводится конференция.
* `start_date`, `end_date` — период проведения.
* `description` — общее описание конференции.
* `participation_terms` — условия участия (тезисы, сбор оргвзноса, требования к докладу и т.п.).

**Особенности:**

* `ordering = ["-start_date"]` — последние (самые свежие) конференции выводятся первыми.
* При удалении места проведения (`Place`) срабатывает `on_delete=models.CASCADE`, то есть все конференции, связанные с этим местом, тоже будут удалены. Это поведение логично, если место является обязательной частью контекста конференции.

Данная модель покрывает часть требований задания: "название конференции, список тематик, место проведения, период проведения, описание, условия участия".

## 4. Модель `Registration`

```
class Registration(models.Model):
    user = models.ForeignKey(User, on_delete=models.CASCADE)
    conference = models.ForeignKey(Conference, on_delete=models.CASCADE)
    presentation_title = models.CharField(max_length=200)
    abstract = models.TextField(blank=True)
    created = models.DateTimeField(auto_now_add=True)
    result = models.CharField(
        max_length=50,
        choices=[
            ('pending', 'Ожидает'),
            ('recommended', 'Рекомендован к публикации'),
            ('not_recommended', 'Не рекомендован')
        ],
        default='pending'
    )

    class Meta:
        constraints = [
            models.UniqueConstraint(
                fields=["user", "conference"],
                name="uniq_user_conference_registration"
            )
        ]

    def __str__(self):
        return f'{self.user.username} - {self.conference.title}'
```

**Назначение.**
`Registration` описывает участие конкретного пользователя в конкретной конференции и его доклад.

Это реализация требований:

* "Регистрация новых пользователей" (пользователь создаёт аккаунт и может регистрироваться);
* "Регистрация авторов для выступлений";
* "Пользователь должен иметь возможность редактирования и удаления своих регистраций".

**Сохраняемые данные:**

* `user` — кто подаёт заявку (ссылка на стандартную модель `User` из `django.contrib.auth`).
* `conference` — на какую конференцию идёт заявка.
* `presentation_title` — название доклада.
* `abstract` — краткие тезисы выступления.
* `created` — дата/время подачи.
* `result` — поле статуса, которое администратор может менять:

  * ожидает проверки,
  * рекомендован к публикации,
  * не рекомендован.

**Особенности:**

* Уникальное ограничение `UniqueConstraint(fields=["user", "conference"])` гарантирует, что один и тот же пользователь не сможет зарегистрироваться на одну и ту же конференцию два раза. Это соответствует бизнес-логике: одна заявка на выступление от одного автора на одну конференцию.
* Наличие поля `result` напрямую выполняет требование лабораторной:

  > "Администратор должен иметь возможность указания результатов выступления (рекомендован к публикации или нет) средствами Django-admin."

То есть админ заходит в Django admin, открывает регистрацию и меняет `result`.

## 5. Модель `Review`

```
class Review(models.Model):
    conference = models.ForeignKey(Conference, on_delete=models.CASCADE)
    user = models.ForeignKey(User, on_delete=models.CASCADE)
    conf_start_date = models.DateField()
    conf_end_date = models.DateField()
    date_posted = models.DateTimeField(auto_now_add=True)
    text = models.TextField()
    rating = models.PositiveIntegerField()

    class Meta:
        unique_together = ['conference', 'user']

    def __str__(self):
        return f'Отзыв от {self.user.username}'
```

**Назначение.**
`Review` — отзыв/комментарий пользователя о конференции.

Это покрывает требование:

> "Написание отзывов к конференциям. При добавлении комментариев должны сохраняться даты конференции, текст комментария, рейтинг (1-10), информация о комментаторе."

**Сохраняемые данные:**

* `conference` — к какой конференции относится отзыв.
* `user` — кто оставил отзыв.
* `conf_start_date`, `conf_end_date` — даты проведения конференции. Они сохраняются явно на момент оставления отзыва, чтобы даже если потом даты в самой конференции изменят, отзыв остался с историческими данными.
* `date_posted` — когда отзыв был оставлен.
* `text` — сам комментарий.
* `rating` — числовой рейтинг (планируется от 1 до 10).

**Особенности:**

* `unique_together = ['conference', 'user']` означает, что один пользователь может оставить только один отзыв на одну конкретную конференцию. Это защищает от спама и накрутки рейтинга.

## Взаимосвязь моделей

Можно описать связи так:

* **Conference** ↔ **Topic**
  Связь многие-ко-многим: одна конференция может относиться к нескольким тематикам, а одна тематика может использоваться многими конференциями.

* **Conference** → **Place**
  Каждая конференция проводится в одном месте (`ForeignKey`), но одно место (`Place`) может использоваться многими конференциями.

* **Registration** связывает `User` и `Conference`.
  Это запись вида «пользователь X заявил доклад Y на конференцию Z», с доп.полями (тезисы, статус).

* **Review** связывает `User` и `Conference`, но не как участника, а как человека, оставившего отзыв и выставившего рейтинг.

---

## Итог

Модели `Topic`, `Place`, `Conference`, `Registration` и `Review` вместе покрывают функциональные требования лабораторной работы:

* хранение полной информации о конференциях (темы, место, период, описание, условия участия);
* возможность регистрации пользователей на выступления;
* возможность редактирования/удаления своих регистраций;
* отдельный рейтинг и отзывы о конференциях;
* поле `result` в `Registration`, которое позволяет администратору через Django admin отмечать, рекомендован ли доклад к публикации.

Таким образом, структура моделей отражает как информационную часть (что за конференция и где она проходит), так и интерактивную часть (кто участвует, кто что сказал, и что об этом думает администратор).
