# Архитектура - Обзор

## Принципы проектирования

Проект построен на основе Clean Architecture с четким разделением на слои и зависимостями, направленными внутрь к domain слою.

## Структура проекта

```
├── cmd/api/              # Точка входа приложения
├── internal/
│   ├── config/           # Конфигурация приложения
│   ├── domain/           # Бизнес-логика (интерфейсы)
│   ├── usecase/          # Сценарии использования
│   ├── infrastructure/   # Реализация (БД, JWT)
│   └── delivery/http/    # HTTP handlers
├── migrations/           # SQL миграции
└── tests/               # Тесты
```

## Слои архитектуры

### Domain Layer

Содержит:
- Интерфейсы репозиториев
- Интерфейсы use cases
- Структуры данных (entities)
- Доменные ошибки

Не зависит от внешних библиотек, что обеспечивает изоляцию бизнес-логики.

### Use Case Layer

Реализует бизнес-логику:
- CRUD операции
- Валидация данных
- Координация между репозиториями

Зависит только от domain интерфейсов, что позволяет легко тестировать и изменять реализацию.

### Infrastructure Layer

Реализует технические детали:
- PostgreSQL репозитории
- JWT сервис
- Bcrypt хеширование паролей
- Логирование

### Delivery Layer

HTTP обработка:
- Chi Router для маршрутизации
- Handlers для обработки запросов
- Middleware для аутентификации, CORS, логирования

## Зависимости

```
Delivery → Use Case → Domain ← Infrastructure
```

Все зависимости направлены внутрь к domain слою, что обеспечивает независимость бизнес-логики от технических деталей.
