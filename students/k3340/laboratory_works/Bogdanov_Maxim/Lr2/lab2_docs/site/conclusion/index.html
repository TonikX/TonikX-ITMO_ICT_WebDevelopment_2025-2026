    Результаты работы

    В ходе выполнения лабораторной работы была разработана полнофункциональная система бронирования отелей на Django с использованием SQLite3.

    Реализованный функционал

    Аутентификация и авторизация: - Регистрация новых пользователей - Вход/выход из системы - Сброс пароля - Разграничение прав доступа

    Управление отелями: - Просмотр списка отелей с поиском - Детальная информация об отелях - Отображение типов номеров и удобств - Таблица постояльцев за последние 30 дней

    Бронирование номеров: - Выбор номера с фильтрацией по множеству параметров - Проверка доступности по датам - Создание, редактирование и отмена бронирований - Валидация дат и предотвращение пересечений - Автоматический расчёт стоимости

    Система отзывов: - Оставление отзывов после заезда - Рейтинг от 1 до 10 - Привязка к конкретным бронированиям - Валидация прав на отзыв - Отображение на странице отеля

    Админ панель: - Управление всеми сущностями системы - Действия заселения и выселения - Валидация переходов статусов

    Дополнительно: - Пагинация всех списков - Фильтрация номеров по 7 параметрам - Средний рейтинг отеля - Адаптивный интерфейс на Bootstrap 5 - Генератор тестовых данных

    Архитектурные решения

    Разделение ответственности

    Проект следует принципу разделения ответственности:

    • models.py - структура данных и валидация на уровне модели
    • services.py - бизнес-логика и сложные запросы
    • forms.py - валидация пользовательского ввода
    • views.py - обработка запросов и связь компонентов
    • templates/ - представление данных

    Такой подход обеспечивает: - Легкость тестирования отдельных компонентов - Переиспользование логики - Упрощение поддержки и масштабирования

    Сложности и их решение

    1. Проверка пересечений интервалов

    Проблема: Необходимо эффективно проверять, не пересекается ли новое бронирование с существующими.

    Решение: Использована логика отрицания непересечения: 

    ~Q(check_out__lte=check_in) & ~Q(check_in__gte=check_out)

    Это проще и понятнее, чем прямая проверка всех случаев пересечения.

    2. Права на оставление отзывов

    Проблема: Нужно проверить, что пользователь действительно проживал в номере.

    Решение: - Привязка отзыва к конкретному бронированию - Проверка статуса брони (checked_in/checked_out) - Валидация даты (можно оставить после заезда) - Запрет дублирования отзывов на одну бронь

    3. Редактирование бронирований

    Проблема: Нельзя разрешить редактирование после заезда или для определённых статусов.

    Решение: Использование свойств модели can_edit и can_cancel: 

    @property
def can_edit(self):
    return (
        date.today() < self.check_in and
        self.status in ['pending', 'confirmed']
    )

    4. Фильтрация с сохранением GET-параметров

    Проблема: При пагинации терялись параметры фильтров.

    Решение: В шаблоне добавлен код для сохранения всех GET-параметров: 

    ?page={{ num }}{% for key, value in request.GET.items %}{% if key != 'page' %}&{{ key }}={{ value }}{% endif %}{% endfor %}

    5. Тестирование с датами в прошлом

    Проблема: Валидация запрещает создавать бронирования с датами в прошлом, но для тестов это нужно.

    Решение: Добавлен параметр skip_validation в метод save(): 

    booking.save(skip_validation=True)
