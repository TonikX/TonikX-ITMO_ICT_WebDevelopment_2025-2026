# Чек-лист разработки Backend Go‑проекта

Чек-лист для разработки Go-проектов на основе лучших практик и типичных замечаний из проверок тестовых заданий (формат Авито‑стажа).

---

## 1. Архитектура и структура проекта

- [ ] Выбрана архитектура (Clean Architecture, Hexagonal, Layered) и она кратко задокументирована
- [ ] Структура проекта соответствует выбранной архитектуре:
  - [ ] `cmd/` — точки входа приложения
  - [ ] `internal/domain/` — доменная модель (entities, value-objects, интерфейсы)
  - [ ] `internal/usecase/` — бизнес-логика (application layer)
  - [ ] `internal/delivery/` — транспортный слой (HTTP, gRPC, CLI, jobs)
  - [ ] `internal/infrastructure/` — адаптеры (БД, внешние сервисы, кэши, брокеры)
- [ ] Бизнес-логика сосредоточена в usecase/сервисах, а не в handlers или репозиториях
- [ ] HTTP-handlers разделены по доменам/ресурсам (нет одного огромного `handlers.go`)
- [ ] Зависимости направлены внутрь (domain не зависит от delivery/infrastructure)
- [ ] Интерфейсы определены в domain/usecase, реализации — в infrastructure
- [ ] Используется dependency injection (wire/fx/ручной DI, без глобальных синглтонов)
- [ ] Используется `context.Context` на границах слоёв (HTTP handlers, репозитории, клиенты)
- [ ] Нет «свалочных» пакетов `utils`, `helpers`, `common` — пакетам даны предметные имена
- [ ] Отдельные пакеты для кросс-срезов: `logger`, `config`, `validator`, `metrics`
- [ ] Отдельная директория/пакет для моков и тестовых реализаций (`mocks/` или `_test.go`), моки не лежат в продакшн‑коде
- [ ] Определены и задокументированы границы модулей/контекстов (если домен достаточно сложный)
- [ ] var _ repository.TeamRepository = (*Repository)(nil)

## 2. Качество кода и линтинг

- [ ] Настроен `golangci-lint` с конфигурационным файлом (`.golangci.yml`)
- [ ] Включены основные линтеры:
  - [ ] `errcheck` — проверка обработки ошибок
  - [ ] `staticcheck` — статический анализ
  - [ ] `govet` — встроенный анализатор Go
  - [ ] `gocritic` — продвинутые правила
  - [ ] `revive` — стиль кода
  - [ ] `gosec` — безопасность
  - [ ] `mnd` — проверка магических чисел
- [ ] Настроены исключения/disable-комментарии и обоснованы (нет «заглушек» без причины)
- [ ] Настроен форматтер (`gofumpt`, `gci`) и он используется в CI
- [ ] Код проходит `go vet`
- [ ] Код отформатирован (`gofmt`/`gofumpt`), импорты отсортированы и сгруппированы (`std` / `external` / `internal`)
- [ ] Все магические числа вынесены в константы, где это повышает читаемость
- [ ] Нет неиспользуемых переменных, функций, мёртвого кода
- [ ] Используются интерфейсы минимальной достаточности (small interfaces)
- [ ] Используются функции стандартной библиотеки вместо самописных велосипедов, где это возможно
- [ ] Не злоупотребляется анонимными `struct{}` — для повторно используемых структур введены именованные типы
- [ ] TODO‑комментарии содержат ссылку на задачу/тикет или понятное пояснение (не просто `// TODO: сделать нормально`)

## 3. Тестирование

- [ ] Настроены unit‑тесты с моками (чётко тестируется бизнес-логика)
- [ ] Настроены интеграционные тесты (с реальной/тестовой БД)
- [ ] Настроены E2E‑тесты (основные пользовательские сценарии)
- [ ] Используется библиотека для генерации моков (`go.uber.org/mock`, `mockery` и т.п.)
- [ ] Покрытие тестами ключевых доменных компонентов ≥ 70%
- [ ] Тесты изолированы и детерминированы (без зависимости от текущего времени, сети и т.п.)
- [ ] Используется отдельная тестовая БД/окружение, тесты не трогают dev/prod-базу
- [ ] Тесты запускаются одной командой (`make test` / `make ci`)
- [ ] Интеграционные/E2E‑тесты используют Docker для изоляции окружения
- [ ] Тесты проверяют и успешные, и ошибочные сценарии
- [ ] Используется табличный стиль unit‑тестов (table-driven tests)
- [ ] Настроен вывод покрытия (`go test -cover`, HTML‑репорт)
- [ ] Для багфиксов добавляются регрессионные тесты

## 4. База данных и схема

- [ ] Используется система миграций (`migrate`, `golang-migrate` и т.п.)
- [ ] Миграции версионированы, имеют `up`/`down` файлы и проверен rollback
- [ ] Стратегия миграций для dev/stage/prod описана и воспроизводима
- [ ] Созданы необходимые индексы (в т.ч. для uniqueness, FK, фильтрации)
- [ ] Ключевые запросы проверены через `EXPLAIN ANALYZE`, индексы добавлены осознанно
- [ ] Для связей «многие‑ко‑многим» используются связующие таблицы, если нужна уникальность пар
- [ ] Часто меняющиеся перечисления (города, типы, статусы) вынесены в справочные таблицы, в основной таблице хранятся `*_id`
- [ ] Бизнес-правила (разрешённые значения, ограничения) проверяются и в коде, а не только CHECK‑constraint’ами
- [ ] Настроены foreign key constraints с корректными `ON DELETE/UPDATE` политиками
- [ ] Используются транзакции для атомарных операций
- [ ] Используется менеджер транзакций / паттерн Unit of Work
- [ ] Избегается N+1 (join’ы, preloading, batch‑операции)
- [ ] Все SQL‑запросы используют параметризованные/prepared statements
- [ ] Динамические части запросов генерируются безопасно (builder, а не конкатенация строк)
- [ ] Настроен connection pooling (`MaxOpenConns`, `MaxIdleConns`, `ConnMaxLifetime`)
- [ ] Настроены таймауты контекста при работе с БД

## 5. Конфигурация

- [ ] Конфигурация вынесена в отдельный пакет/модуль
- [ ] Поддерживается загрузка конфигурации через файлы (YAML/JSON/TOML)
- [ ] Поддерживается переопределение через переменные окружения
- [ ] Переменные окружения имеют префикс приложения (например, `APP_`)
- [ ] Выполняется валидация конфигурации при загрузке (обязательные поля, диапазоны)
- [ ] Установлены разумные значения по умолчанию
- [ ] Разные конфиги для разных окружений (development, docker, e2e, staging, production)
- [ ] Чувствительные данные (DSN БД, JWT‑секрет, ключи) берутся только из env/secret-менеджера
- [ ] В репозитории нет `.env` с реальными секретами, есть только `.env.example`
- [ ] В конфиге разделены:
  - [ ] HTTP-сервер
  - [ ] БД
  - [ ] Внешние сервисы
  - [ ] Логирование
  - [ ] Метрики/трейсинг
- [ ] Фичефлаги могут настраиваться через конфиг/флаги

## 6. Docker и контейнеризация

- [ ] Создан `Dockerfile` с multi-stage build (маленький финальный образ, non-root user)
- [ ] Используется `.dockerignore`
- [ ] Создан `docker-compose.yml` для локальной разработки
- [ ] Есть отдельные compose‑файлы/профили для тестов (integration, e2e)
- [ ] Контейнеры имеют настроенные healthchecks (БД, приложение, брокеры)
- [ ] Сервис слушает на `0.0.0.0`, порт задаётся конфигом/ENV
- [ ] Настроены `volumes` для персистентности данных там, где это нужно
- [ ] Настроены `networks` для изоляции сервисов
- [ ] Миграции применяются автоматически при старте контейнера или отдельным job’ом
- [ ] Переменные окружения передаются через compose/секрет‑файлы
- [ ] Версия Go в `go.mod` и в Dockerfile синхронизированы
- [ ] Проект поднимается «с нуля» одной командой из README (`docker compose up` / `make up`)

- [ ] CI собирает бинарный файл/контейнерный образ
- [ ] Используется кеширование модулей Go и build cache
- [ ] Используется версия Go из `go.mod`
- [ ] Есть агрегирующая команда (`make ci` / `make check`), используемая и локально, и в CI

## 9. Безопасность

- [ ] Все SQL‑запросы используют параметризованные запросы (нет SQL injection)
- [ ] Есть валидация входных данных на уровне handlers/usecase
- [ ] Ограничен размер тела запроса (MaxBodySize)
- [ ] Ограничен размер заголовков (MaxHeaderBytes)
- [ ] Используется `gosec` в CI
- [ ] Секреты не коммитятся в репозиторий
- [ ] Настроен `.gitignore` для исключения чувствительных файлов
- [ ] При наличии паролей они хранятся только в виде безопасных хэшей (bcrypt/argon2)
- [ ] Токены, пароли и другие секретные данные не логируются и не попадают в ответы клиенту
- [ ] Используются стандартные заголовки авторизации: `Authorization: Bearer <token>`
- [ ] Для внешних HTTP‑запросов есть таймауты и ограничители (rate limit, retries)
- [ ] TLS/HTTPS: задокументировано, кто терминирует TLS (ingress/proxy/app)

- [ ] Оптимизированы SQL‑запросы (индексы, планы, избегание N+1)
- [ ] Используются batch‑операции для массовых обновлений/инсертов
- [ ] Настроен connection pooling
- [ ] Проведено нагрузочное тестирование (K6, JMeter, Locust, Apache Bench и т.п.)

- [ ] Результаты нагрузочного тестирования соответствуют SLI (или задокументированы отклонения)

- [ ] Используются профайлеры (`pprof`, `go tool pprof`) для поиска bottleneck’ов

## 11. HTTP‑сервер и API

- [ ] Настроены таймауты (`ReadTimeout`, `WriteTimeout`, `IdleTimeout`, `ReadHeaderTimeout`)
- [ ] Настроен graceful shutdown (context с таймаутом, ожидание завершения запросов)
- [ ] Реализованы healthcheck‑эндпоинты (`/healthz`/`/live`/`/ready`)
- [ ] Разделены liveness и readiness‑проверки (для orchestrator’а)
- [ ] Healthcheck поддерживает минимум GET (HEAD — опционально)
- [ ] Используются middleware для:
  - [ ] логирования запросов
  - [ ] recovery от паник
  - [ ] Request ID для трейсинга
  - [ ] ограничения размера тела запроса
  - [ ] rate limiting (если требуется)
- [ ] Структурированное логирование запросов
- [ ] Логи содержат `request_id`, `user_id` (где релевантно), `trace_id`
- [ ] Реализация строго совпадает со спецификацией: путь, метод, статусы, формат тела ответа, `Content-Type`
- [ ] Заголовок авторизации читается из `Authorization: Bearer <token>`

## 12. Обработка ошибок

- [ ] Определены доменные ошибки в `internal/domain/.../errors.go` (или аналогичной структуре)
- [ ] Ошибки маппятся на HTTP‑статусы (и/или gRPC‑коды) через единый слой
- [ ] Единый формат ответов об ошибках (код, сообщение, детали)
- [ ] Внешнему клиенту не возвращаются сырые ошибки БД/инфраструктуры (stack trace, SQL‑сообщения)
- [ ] Ошибки логируются с контекстом (ключевые поля, идентификаторы)
- [ ] Для проверки ошибок используется `errors.Is` и `errors.As`, а не `==` / `strings.Contains`
- [ ] Нет «проглоченных» ошибок: каждая либо обработана, либо явно проигнорирована с объяснением
- [ ] Есть разделение на user‑facing и internal‑ошибки

## 14. Логирование и метрики

- [ ] Используется структурированное логирование (`slog`, `zap`, `zerolog`, `logrus`)
- [ ] Логи в JSON‑формате в продакшене, читабельный формат в dev
- [ ] Логи содержат уровни (`debug`, `info`, `warn`, `error`)
- [ ] Логи содержат контекст (`request_id`, `user_id`, `trace_id`, ключевые параметры)
- [ ] Ошибки логируются не только на уровне HTTP, но и в бизнес‑логике
- [ ] Добавлены метрики (Prometheus/OpenTelemetry):
  - [ ] latency/длительность запросов
  - [ ] количество запросов по статус‑кодам
  - [ ] количество ошибок по типам
  - [ ] ключевые бизнес‑метрики (кол-во заказов, регистраций и т.п.)
- [ ] Есть базовые дашборды (latency, errors, RPS, ресурсы)
- [ ] Не логируются чувствительные данные (пароли, токены, личная информация)

## 15. Транзакции

- [ ] Используется менеджер транзакций/Unit of Work для атомарных операций
- [ ] Транзакции используются только там, где действительно есть несколько связанных изменений
- [ ] Простые запросы не оборачиваются в транзакции «на всякий случай»
- [ ] Нет неоправданно долгих транзакций (нет долгих RPC/IO внутри транзакции)
- [ ] Правильная обработка rollback при ошибках (`defer` + проверка `err`)
- [ ] Решение об использовании транзакции принимается на уровне usecase/сервиса, а репозитории умеют работать как с соединением, так и с транзакцией

---

## 16. Валидация

- [ ] Валидация входных данных вынесена в отдельный пакет/слой
- [ ] Валидация выполняется до бизнес‑логики
- [ ] Валидация проверяет типы, диапазоны, обязательность полей
- [ ] Сообщения об ошибках валидации понятны пользователю и, при необходимости, локализуемы
- [ ] Проверяются бизнес‑правила: права пользователя, запрет self‑операций, принадлежность сущности и т.п.
- [ ] Бизнес‑ограничения по спискам значений (города, типы, статусы) описаны в коде/конфиге, а не только в БД
- [ ] Валидация не дублируется без необходимости (переиспользуются общие функции/структуры)

## 20. Аутентификация и авторизация

- [ ] Определена стратегия аутентификации (JWT/OAuth2/session и т.п.)
- [ ] Токены/сессии безопасно хранятся (HTTP‑only cookies, `secure`, разумный TTL)
- [ ] Роли и права доступа (RBAC/ABAC) явно описаны и реализованы
- [ ] Проверка прав доступа централизована (middleware/policy layer)
- [ ] Критические операции защищены дополнительными проверками (2FA, подтверждения)
- [ ] Пароли/секреты никогда не логируются и не попадают в ответы API
- [ ] Используется стандартный формат `Authorization: Bearer <token>`

---

## 21. Наблюдаемость и эксплуатация

- [ ] Настроен трейсинг (OpenTelemetry/Jaeger/Tempo и т.п.)
- [ ] Request ID/Trace ID прокидываются через все слои и во внешние вызовы
- [ ] Настроены алерты по ключевым метрикам (latency, error rate, saturation, ресурсы)
- [ ] Есть базовые runbook’и: что делать при деградации/отказе сервиса
- [ ] Логи, метрики и трейсинг интегрированы по `trace_id`/`span_id`
- [ ] Есть стратегия резервного копирования БД и проверено восстановление
- [ ] Описан процесс zero‑downtime deployment (или задокументирован допустимый downtime)
