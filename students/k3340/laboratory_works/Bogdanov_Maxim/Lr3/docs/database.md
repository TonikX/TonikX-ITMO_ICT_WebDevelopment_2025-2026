# Схема базы данных для школьной системы

## 1. Описание модели данных

Схема построена на следующих основных сущностях:

### Справочники:
- **academic_years** — учебные годы (централизованный справочник)
- **subjects** — предметы (математика, физика и т.д.)
- **subject_types** — типы дисциплин (базовая/профильная)
- **genders** — справочник полов
- **weekdays** — дни недели
- **grading_periods** — оценочные периоды (четверти) с привязкой к учебному году

### Основные сущности:
- **teachers** — учителя
- **teacher_subjects** — какие предметы преподает каждый учитель (многие-ко-многим)
- **classrooms** — кабинеты с указанием типа дисциплины
- **classes** — классы (9А, 10Б и т.д.) с указанием классного руководителя и учебного года
- **students** — ученики с привязкой к классу
- **schedule** — расписание занятий
- **grades** — оценки учеников по предметам в разрезе четвертей

### Связи:
- Учитель ↔ Предметы (многие-ко-многим через teacher_subjects)
- Учитель → Кабинет (многие-к-одному, nullable) — несколько учителей могут использовать один кабинет
- Класс → Учитель-классный руководитель (многие-к-одному)
- Класс → Учебный год (многие-к-одному)
- Ученик → Класс (многие-к-одному)
- Расписание связывает Класс + День + Урок → Предмет + Учитель + Кабинет
- Оценки связывают Ученика + Предмет + Четверть → Оценка

## 3. Стратегия удаления и индексы

### Soft Delete vs Hard Delete:

**Таблицы с Soft Delete:**
- **teachers** — сохраняем историю уволившихся учителей для целостности архивных данных (расписание, оценки прошлых лет)
- **students** — сохраняем данные отчисленных учеников для архива успеваемости и соблюдения требований хранения персональных данных
- **classes** — классы прошлых лет остаются в архиве, но помечаются как удаленные

**Обоснование:** Образовательные данные имеют юридическую и историческую ценность. Удаление учителя или ученика не должно разрушать связность данных в отчетах за прошлые периоды. Soft delete позволяет сохранить целостность данных и при необходимости восстановить записи.

**Таблицы с Hard Delete:**
- **Все справочники** (academic_years, subjects, subject_types, genders, weekdays, grading_periods) — стабильные данные, редко удаляются
- **classrooms** — при удалении кабинета все связи должны быть пересмотрены
- **teacher_subjects** — каскадное удаление при удалении учителя; исторические данные контролируются через поля start_date/end_date
- **schedule** — расписание привязано к классу и может меняться; при архивации класса расписание сохраняется через связь
- **grades** — оценки не удаляются в нормальном сценарии; при необходимости удаления ученика оценки остаются через FK

### Ключевые индексы и их назначение:

**Основные запросы завуча:**

1. **idx_schedule_class_weekday_lesson** — для запроса "какой предмет в классе X в день Y на уроке Z"
  - Покрывает основной запрос расписания полностью

2. **idx_grades_student_period** — для формирования отчета об успеваемости класса за четверть
  - Позволяет быстро получить все оценки ученика за конкретный период

3. **idx_students_class_gender** — для подсчета мальчиков/девочек в каждом классе
  - Композитный индекс для агрегации по полу в рамках класса

4. **idx_teacher_subjects_subject** — для запроса "учителя, преподающие те же предметы"
  - Partial index только для активных назначений (end_date IS NULL)

5. **idx_schedule_teacher, idx_schedule_subject, idx_schedule_classroom** — для различных срезов расписания
  - Поиск всех уроков учителя, предмета или в конкретном кабинете

**Вспомогательные индексы:**

6. **idx_students_class** — для списка учеников класса (без учета пола)
7. **idx_classes_academic_year** — для выборки классов текущего года
8. **idx_classes_teacher** — для поиска классов конкретного классного руководителя
9. **idx_grading_periods_academic_year** — для выборки четвертей учебного года
10. **idx_subjects_type, idx_classrooms_type** — для подсчета кабинетов по типам дисциплин

**Почему убраны бессмысленные индексы:**
- Удалены все композитные индексы, включающие `id` в качестве первого или единственного значимого поля
- PRIMARY KEY автоматически создает B-tree индекс на `id`, поэтому дублировать его не нужно
- Partial indexes теперь фильтруют только по `deleted_at` и другим условиям, не включая `id` в состав индекса

Схема полностью поддерживает все операции завуча и информационные запросы из требований.
