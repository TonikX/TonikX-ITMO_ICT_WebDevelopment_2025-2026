# Задание 1 — Суммирование `1..N`

## Постановка

Посчитать сумму всех чисел от 1 до `N` тремя способами: `threading`, `multiprocessing`, `asyncio`. Задача разбивается на несколько подзадач, которые запускаются параллельно; затем замеряется время и сравниваются результаты.

ТЗ предлагает `N = 10**13`, однако чистый Python-цикл такой длины занял бы десятки минут и не помогал бы увидеть разницу подходов. Взято `N = 10**8`, `WORKERS = 4` — этого достаточно, чтобы отчётливо проявился эффект GIL.

## Реализация

У каждого варианта есть функция `calculate_sum(...)`, которая считает сумму в своём диапазоне. Главная функция режет `[1, N]` на `WORKERS` равных кусков и запускает их параллельно.

### `threading_sum.py`

```python
def calculate_sum(start, end, out, idx):
    s = 0
    for i in range(start, end + 1):
        s += i
    out[idx] = s
```

Потоки стартуют через `threading.Thread`, результаты собираются в список `results`.

### `multiprocessing_sum.py`

```python
def calculate_sum(bounds):
    start, end = bounds
    return sum(range(start, end + 1))

with mp.Pool(processes=WORKERS) as pool:
    parts = pool.map(calculate_sum, ranges)
```

### `async_sum.py`

```python
async def calculate_sum(start, end):
    s = 0
    for i in range(start, end + 1):
        s += i
    return s

parts = await asyncio.gather(*tasks)
```

## Результаты

Замер на Apple M-серии, Python 3, `N = 10**8`, 4 воркера (`task1/benchmark.py`):

| Подход            | Сумма                  | Время, с |
|-------------------|------------------------|----------|
| `threading`       | 5 000 000 050 000 000  | 3.106    |
| `multiprocessing` | 5 000 000 050 000 000  | **1.022** |
| `asyncio`         | 5 000 000 050 000 000  | 3.292    |

## Анализ

- **`threading` ≈ `asyncio`.** GIL сериализует исполнение Python-байткода — несмотря на 4 «параллельных» воркера, одновременно считает только один. Для CPU-bound кода оба подхода дают то же время, что и последовательный цикл (плюс накладные расходы).
- **`multiprocessing` в ~3× быстрее.** Каждый воркер — отдельный процесс с собственным интерпретатором и GIL, CPU-время честно раскладывается на ядра. Ускорение не ровно 4× из-за стоимости `fork`/pickle/сборки результата.
- **Вывод.** Для CPU-bound задач в Python единственный рабочий способ получить параллелизм — `multiprocessing` (или вынос кода в C-расширение, которое отпускает GIL).
