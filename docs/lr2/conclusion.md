# Выводы

## Результаты работы

В рамках лабораторной работы было разработано полнофункциональное веб-приложение "Табло победителей автогонок" на фреймворке Django. Приложение позволяет организовывать автомобильные соревнования, регистрировать участников и отображать результаты заездов.

## Выполненные задачи

### 1. Изучение Django

В процессе работы я изучила основные концепции фреймворка Django:

- Архитектуру MVT (Model-View-Template)
- Работу с ORM (Object-Relational Mapping)
- Систему маршрутизации URL
- Работу с шаблонами
- Систему форм и валидации
- Встроенную админ-панель

### 2. Разработка моделей данных

Были созданы модели для представления предметной области:

- Team - команды гонщиков
- DriverProfile - профили водителей
- Race - информация о гонках
- Heat - заезды в рамках гонок
- HeatResult - результаты водителей
- Registration - регистрации на гонки
- Comment - комментарии пользователей

Модели связаны между собой различными типами отношений (один-к-одному, один-ко-многим). Реализованы ограничения уникальности и валидация данных на уровне базы данных.

### 3. Реализация бизнес-логики

Были реализованы основные бизнес-процессы:

- Регистрация пользователей с автоматическим созданием профиля
- Обязательное заполнение профиля перед участием
- Регистрация на гонки с проверкой уникальности
- Добавление результатов с валидацией регистрации водителя
- Система комментариев с рейтингами
- Разделение прав доступа между пользователями и администраторами

### 4. Создание пользовательского интерфейса

Интерфейс разработан с использованием Bootstrap 5:

- Адаптивный дизайн для разных устройств
- Навигационное меню с разными вариантами для разных типов пользователей
- Карточки для отображения гонок
- Таблицы для результатов заездов и списков участников
- Формы с валидацией
- Пагинация длинных списков
- Поиск и сортировка данных
- Информативные сообщения о результатах операций

### 5. Настройка админ-панели

Админ-панель Django была настроена для удобного управления:

- Списки с фильтрами и поиском
- Inline формы для связанных объектов
- Кастомные действия (экспорт в CSV, массовая публикация)
- Счетчики связанных объектов
- Ограничение выбора водителей в результатах

## Трудности и их решение

### 1. Связи между моделями

**Проблема**: Сначала было непонятно, как правильно организовать связи между моделями, особенно между Heat, Registration и HeatResult.

**Решение**: После изучения документации и нескольких итераций была разработана правильная структура, где результаты связаны и с заездом, и с водителем, а регистрация проверяется через валидацию.

### 2. Валидация регистрации водителей

**Проблема**: Нужно было обеспечить, чтобы результаты могли быть добавлены только для водителей, зарегистрированных на гонку.

**Решение**: Реализована кастомная валидация в методе `clean()` модели HeatResult, которая проверяет наличие активной регистрации перед сохранением.

### 3. Обязательное заполнение профиля

**Проблема**: Профиль создается автоматически пустым, нужно было заставить пользователей заполнить его перед участием.

**Решение**: Добавлены проверки в представлениях регистрации на гонку и добавления комментариев, которые перенаправляют на заполнение профиля.

### 4. Пагинация с сохранением параметров

**Проблема**: При переходе между страницами терялись параметры поиска и сортировки.

**Решение**: Реализован механизм сохранения query параметров через `urlencode()` и передачу их в шаблон.

### 5. Разделение прав админов и пользователей

**Проблема**: Нужно было показывать разную информацию и функционал для админов и обычных пользователей.

**Решение**: Создан флаг `is_admin` в контексте шаблонов и условная логика в представлениях и шаблонах.
