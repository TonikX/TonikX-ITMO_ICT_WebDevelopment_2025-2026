# Структура проекта

## Общая организация

Проект построен на фреймворке Django и следует его стандартной структуре. Django использует паттерн MVT (Model-View-Template), который похож на MVC, но немного отличается.

## Как устроен Django MVT

### Models (Модели)

Модели - это классы Python, которые описывают структуру данных в базе. Django автоматически создает таблицы в базе данных на основе этих классов.

В моем проекте есть такие модели:

- **User** - встроенная модель Django для пользователей
- **Team** - команды гонщиков
- **DriverProfile** - профиль водителя (связан с User)
- **Race** - информация о гонках
- **Heat** - заезды в рамках гонки
- **HeatResult** - результаты водителей в заезде
- **Registration** - регистрация водителя на гонку
- **Comment** - комментарии к гонкам

### Views (Представления)

Представления - это функции или классы, которые обрабатывают запросы от пользователя и возвращают ответ (обычно HTML страницу).

Я использовала два типа представлений:

**Class-Based Views (CBV)** - для стандартных операций:

- ListView - для списков объектов
- DetailView - для просмотра одного объекта
- CreateView - для создания нового объекта
- UpdateView - для редактирования
- DeleteView - для удаления

**Function-Based Views (FBV)** - для специфичной логики:

- register_for_race - регистрация на гонку
- unregister_from_race - отмена регистрации
- add_comment - добавление комментария
- signup - регистрация нового пользователя

### Templates (Шаблоны)

Шаблоны - это HTML файлы с дополнительным синтаксисом Django для вставки данных и логики. Они определяют, как будет выглядеть страница для пользователя.

Используется базовый шаблон `base.html`, от которого наследуются все остальные страницы. Это позволяет не дублировать код меню, футера и других общих элементов.

## Как работает запрос

Когда пользователь открывает страницу, происходит следующее:

1. **URL** - Django смотрит в urls.py и определяет, какое представление должно обработать запрос
2. **View** - Представление получает данные из базы через модели
3. **Template** - Представление передает данные в шаблон
4. **Response** - Шаблон рендерится в HTML и отправляется пользователю

Пример: пользователь открывает список гонок

```
Запрос: /
    ↓
urls.py находит путь '' → вызывает RaceListView
    ↓
RaceListView получает список гонок из базы
    ↓
Передает данные в шаблон race_list.html
    ↓
Возвращает готовую HTML страницу пользователю
```

## Миксины и переиспользуемый код

Для избежания дублирования кода я создала кастомный миксин:

**OwnerRequiredMixin** - проверяет, что пользователь является владельцем объекта или администратором. Используется для редактирования и удаления объектов.

Пример использования:

```python
class RegistrationUpdateView(LoginRequiredMixin, OwnerRequiredMixin, UpdateView):
    # Только владелец регистрации или админ может ее редактировать
    model = Registration
    ...
```

## Сигналы Django

Сигналы позволяют выполнять код автоматически при определенных событиях.

Я использовала сигнал `post_save` для автоматического создания профиля водителя при регистрации нового пользователя:

```python
@receiver(post_save, sender=User)
def create_or_update_driver_profile(sender, instance, created, **kwargs):
    if created:
        DriverProfile.objects.create(user=instance, full_name='')
```

Теперь при создании пользователя автоматически создается пустой профиль водителя, который нужно заполнить.

## Миграции

Миграции - это способ Django применять изменения в моделях к базе данных. При изменении моделей нужно:

1. Создать миграцию: `python manage.py makemigrations`
2. Применить ее: `python manage.py migrate`

Django автоматически генерирует SQL запросы для создания и изменения таблиц.
